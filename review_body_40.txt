## FE PR #40 코드 리뷰 — 관리자 콘솔 페이지 MVP

### 잘 된 부분

- `Promise.all` + `cancelled` 플래그로 언마운트 시 state 업데이트 차단 패턴이 모든 fetch에 일관되게 적용됨
- `safeJson(res)` 헬퍼로 개별 API 실패를 null 처리 → 대시보드 전체가 깨지지 않는 설계
- `confirmMode` 2단계 확인(승인/거절) UX, ESC 키 닫기, 오버레이 클릭 닫기 양쪽 구현
- CSS 전용 막대 차트로 외부 라이브러리 없이 번들 크기 절약
- URL `?menu=` 파라미터로 메뉴 상태 유지 — 새로고침·공유 가능

### 🔴 Auth guard가 `useEffect` 안에 있어 비ADMIN도 admin API를 호출함

**`AdminPage.jsx` 약 line 322–350**

```jsx
// 둘 다 첫 렌더 직후 동시에 실행됨
useEffect(() => {
  if (!getAccessToken() || getRole() !== 'ADMIN') navigate('/', { replace: true })
}, [navigate])

useEffect(() => {
  Promise.all([
    authFetch(`.../user-count`),
    authFetch(`.../user-count/inactive`),
    // ...5개 admin 엔드포인트
  ])
}, [])
```

두 `useEffect`는 첫 렌더 이후 동시에 실행됩니다. 즉, 비ADMIN이 `/admin`을 직접 입력하면 리다이렉트 전에 이미 admin API 5개가 서버로 나갑니다.

또한 페이지 새로고침 시 토큰 재발급(`reissueAccessToken`)이 완료되기 전에 `getAccessToken()`이 null을 반환하여 정상 ADMIN 사용자도 홈으로 튕길 수 있습니다(`tokenLoading`이 true인 구간).

**권장 수정:** `tokenLoading`이 완료될 때까지 렌더링을 보류한 후 동기적으로 역할 체크

```jsx
// tokenStore.js에 getTokenLoading() export 필요
const [tokenReady, setTokenReady] = useState(!getTokenLoading())

useEffect(() => {
  if (getTokenLoading()) {
    const handler = (e) => {
      if (!e.detail.tokenLoading) {
        setTokenReady(true)
        window.removeEventListener('accessTokenChanged', handler)
      }
    }
    window.addEventListener('accessTokenChanged', handler)
    return () => window.removeEventListener('accessTokenChanged', handler)
  }
}, [])

if (!tokenReady) return <div className="admin-loading">...</div>
if (!getAccessToken() || getRole() !== 'ADMIN') return <Navigate to="/" replace />
```

이렇게 하면 권한 확인 전에 API가 나가는 문제가 없어지고, 새로고침 후 정상 ADMIN이 홈으로 튕기는 문제도 함께 해결됩니다.

### 🟠 `setAccessToken` → `setUserInfo` 순서로 `accessTokenChanged` 이벤트가 2번 발생

**`authApi.js` line 58–59 / `LoginPage.jsx` / `AuthBootstrap.jsx`**

```js
setAccessToken(data.accessToken, data.accessExpiresIn)
// 이벤트 1: name=null, role=null 상태로 발송 → Navbar userName이 잠깐 null
setUserInfo({ name: data.name, role: data.role, userId: data.userId })
// 이벤트 2: 정상 값으로 발송
```

로그인 직후 Navbar에서 `userName`이 null → 이름 없이 렌더 → 즉시 재렌더 되는 깜빡임이 발생합니다.

**수정:** `setAccessToken`에 userInfo를 함께 받거나, 두 호출을 묶는 `setAuthData(token, expiresIn, { name, role, userId })` 함수를 만들어 이벤트를 1회만 발송하세요.

### 🟠 거절 사유를 query param으로 전송 — 서버·프록시 로그에 평문 노출

**`VerificationDetailModal.handleReject` 약 line 989–992**

```js
const rejectUrl = new URL(`${API_BASE_URL}/.../reject`)
if (rejectReason.trim()) rejectUrl.searchParams.set('rejectReason', rejectReason.trim())
authFetch(rejectUrl.toString(), { method: 'PATCH' })
```

거절 사유가 URL 쿼리 파라미터로 전송되면 Nginx / CloudFront access log에 평문으로 기록됩니다. request body JSON으로 보내도록 변경하세요 (BE API도 query param → body로 함께 수정 필요):

```js
authFetch(`${API_BASE_URL}/.../reject`, {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ rejectReason: rejectReason.trim() || null }),
})
```

### 🟠 페이지네이션 — 전체 페이지 수만큼 DOM 노드 렌더

**`TeacherApprovalPanel` 약 line 879 / `UserManagementPanel` 약 line 1321**

```jsx
{Array.from({ length: totalPages }, (_, i) => (
  <div key={i} className={`page-num...`}>{i + 1}</div>
))}
```

회원이 수천 명이면 totalPages가 수백이 되어 버튼도 수백 개 렌더됩니다. 현재 페이지 ±2 범위만 보여주는 방식(예: `< 1 … 4 5 6 … 30 >`)으로 제한하세요.

### 🟡 `WeeklyStatChart` — `dates`가 매 렌더마다 재계산되어 eslint 규칙 억제

**`WeeklyStatChart` 약 line 508, 542**

```jsx
const dates = Array.from({ length: 7 }, (_, i) => toDateStr(daysAgo(7 - i)))
// ...
}, []) // eslint-disable-line react-hooks/exhaustive-deps
```

`dates`가 렌더마다 새 배열 참조라 deps에 포함하면 무한 루프, 빼면 stale → lint 규칙을 억제합니다. `useMemo`로 한 번만 계산하면 경고 없이 deps에 포함할 수 있습니다:

```jsx
const dates = useMemo(
  () => Array.from({ length: 7 }, (_, i) => toDateStr(daysAgo(7 - i))),
  []
)
```
